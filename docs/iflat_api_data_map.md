# Карта данных API CRM (Базис.Недвижимость / YOUR_CRM_API_HOST)

Документ составлен по фактическому использованию в проекте (`Rooms_Refresh`, `Acceptance_Refresh`, `parse_rooms`, `parse_inspections`, `template_b.py`, `template_a.py`) и по публичному описанию API на сайте.  
**Полная Swagger-схема** в интерфейсе: [настройки интеграции](https://YOUR_CRM_API_HOST/panel/settings/own_integration) ([описание возможностей API](https://YOUR_CRM_API_HOST/static/about-api)).

**Важно про доступ к Swagger и OpenAPI:** токен из `POST /api/v1/oauth/token` (Bearer) **не** подходит для HTML Swagger UI и для JSON спецификации — сервер отвечает «Unauthenticated». Нужна **веб-сессия**: один раз залогиниться как в браузере (`POST https://YOUR_CRM_API_HOST/login` с полями `_token`, `phone`, `password`, `remember`), затем с тем же `requests.Session` открыть [`GET https://YOUR_CRM_API_HOST/docs/api-docs.json`](https://YOUR_CRM_API_HOST/docs/api-docs.json) — там полный OpenAPI **3.0**, версия спецификации на момент выгрузки **1.4.0**, **261** путь. Офлайн-снимок сохранён в репозитории: **`iflat_openapi_1.4.0.json`** (~1.5 МБ). Полный перечень всех операций из этого снимка (путь × метод × `operationId` × описание × теги): **`iflat_swagger_all_operations.md`**. Файл `service-account.json` — это **Google service account** для Sheets, не учётные данные CRM.

## 2. Авторизация OAuth

**Эндпоинт:** `POST https://YOUR_CRM_API_HOST/api/v1/oauth/token`

**Тело запроса (form / как в скриптах):**

| Поле | Назначение |
|------|------------|
| `username` | Логин пользователя CRM |
| `password` | Пароль |
| `account_id` | Идентификатор аккаунта |
| `client_id` | OAuth client |
| `client_secret` | Секрет клиента |
| `grant_type` | В проектах: `login` |

**Ответ:** JSON с полем `access_token` (используется во всех последующих GET).

**Откуда брать данные:** только из ответа токена и последующих ресурсов; секреты хранить вне кода (env, файл ключей не в git).

## 3. Помещения: `GET /api/v1/rooms`

Источник основной строки по каждому объекту (квартира, кладовая, машино-место, офис и т.д.).

### 3.1. Параметры запроса (из кода)

| Параметр | Где используется | Смысл |
|----------|------------------|--------|
| `page` | Везде | Номер страницы |
| `perPage` | Обычно `100` | Размер страницы |
| `embed` | `room_type,status,floor,house,deal` | Подгрузка связанных сущностей одним запросом |
| `orderBy` | Например `-updated_at` | Сортировка |
| `statusId` | `Rooms_Refresh`, `parse_rooms` | Фильтр по статусу помещения (см. §6) |
| `houseId` | `template_b`, `template_a` | Ограничение по дому |
| `roomType` | Числовой ID типа помещения (напр. квартира `1`, кладовая `5`) |
| `saleStatuses` | Например `FREE` | Для подсчёта непроданных (`meta.total`) |

Дополнительно в коде не использовались, но типичны для REST: фильтры по датам обновления и т.д. — уточнять в Swagger в ЛК.

### 3.2. Поля объекта `room` (фактически читаемые в проекте)

Верхний уровень:

| Поле / путь | Откуда | Назначение |
|-------------|--------|------------|
| `id` | Корень `room` | **Ключ комнаты в CRM** — лучший ключ для связки с приёмками и отчётами |
| `house_id` | Корень | Связь с домом |
| `status_id` | Корень | Числовой статус передачи / приёмки объекта |
| `room_type_id` | Корень | Тип помещения (квартира/кладовая/…) |
| `take_date_from` | Корень | Начало окна клиентской приёмки (ISO-дата в JSON) |
| `take_date_to` | Корень | Конец окна приёмки |
| `deal` | Вложенный объект при `embed` содержит `deal` | Сделка по помещению |
| `house` | Вложенный при `embed` | Дом |
| `room_type` | Вложенный при `embed` | Справочник типа помещения |
| `status` | Вложенный при `embed` | Справочник статуса (имя и др.) |
| `floor` | Запрашивается в `embed`, в выгрузках в таблицу напрямую не писали | Этаж и связанная геометрия — см. Swagger |

## 4. Клиентские приёмки (осмотры): `GET /api/v1/inspections`

Источник событий «приёмка», счётчиков визитов, замечаний, пользователя.

### 4.0. Тип приёмки (`typeId` в запросе, `type_id` в объекте) — не путать со статусом

По официальному описанию API (OpenAPI) значения **типа** фиксированы:

| `typeId` / `type_id` | Название |
|----------------------|----------|
| **1** | Клиентская приёмка |
| **2** | Внутренняя приёмка |
| **3** | Передача управляющей компании |

В коде вместо цифр удобнее использовать имена из модуля **`iflat_inspection_labels.py`**:  
`inspection_type_id("client")` → `1`, `inspection_type_label(1)` → «Клиентская приёмка».

**Важно:** это **не** `status_id` приёмки (этап «в процессе / принято» — свой справочник `GET /inspections/statuses`) и **не** `status_id` **помещения** в `/rooms` (принято / с замечаниями / односторонняя передача — см. §5).

### 4.1. Параметры запроса (из кода)

| Параметр | Пример | Смысл |
|----------|--------|--------|
| `page`, `perPage` | Пагинация | |
| `typeId` | `1` (= клиентская) | **Тип** приёмки: см. таблицу §4.0 |
| `statusId` | `4`, `8`, и др. | Статус записи приёмки |
| `takeDateFrom`, `takeDateTo` | `ДД.MM.YYYY` | Окно по датам приёмки (в `Acceptance_Refresh` / `parse_inspections`) |
| `houseId` | Число | Все приёмки по дому (`template_b`) |
| `roomId` | Список id через запятую (чанки по 50) | Приёмки по списку комнат + `embed=claims` |
| `embed` | `status`, `room`, `room.house`, `room.room_type`, `user`, `claims`, … | Связанные данные |

### 4.2. Поля объекта `inspection` (читаемые в проекте)

| Поле | Назначение |
|------|------------|
| `id` | ID записи приёмки |
| `room_id` | Связь с `rooms.id` |
| `status_id` | Статус процесса приёмки |
| `updated_at`, `created_at` | Время обновления/создания (строка ISO) |
| `take_date_from`, `take_date_to` | Окно записи (если отдаёт API — добавлено в расширенном `Acceptance_Refresh`) |
| `room` | Вложенный объект помещения (частично дублирует `/rooms`) |
| `user` / `inspector` | Ответственный / исполнитель (обработка разными ключами в коде) |
| `status` | Вложенный справочник статуса |
| `claims` | Массив замечаний (при `embed=claims`) |

### 4.3. Объект `claim` (замечание)

Использование в `template_b`: `claim["status_id"]` для классификации «в работе» vs «готово».

| Константы в коде | Смысл (локальная бизнес-логика) |
|------------------|----------------------------------|
| `claims_inwork_statuses = [1, 2, 8]` | Замечания в работе |
| `claims_done_statuses = [9, 3]` | Замечания закрыты |

Точные названия статусов — в Swagger / ответе API.

## 7. Связка данных между системами

### 7.1. Внутри CRM

| От | К | Ключ |
|----|---|------|
| `rooms.id` | `inspections.room_id` | ID комнаты |
| `rooms.house_id` | `house.id` | Дом |
| `rooms.deal` | — | Одна сделка на помещение в типичном сценарии |

### 7.2. Google Таблица «CRM» и лист «Импорт»

| Источник | Поле | Назначение |
|----------|------|------------|
| CRM | `rooms.id` | Уникальный идентификатор строки «Комнаты» (колонка с ID комнаты) |
| Таблица | Колонка «Код» / шифр объекта | Связка с листом **Импорт** (`ШИФР ОБЪЕКТА`) для проекта, фазы, корпуса, объёмов КВ/КЛ/ММ/БКФН, отделки |
| Импорт | Плановые даты, объёмы | **Не из API Кометы** — загружаются из вашего бизнес-контура |

Итог: **полный срез для Power BI** = факт из CRM (`rooms` + `inspections`) **Merge** по `room id` и **Merge** с Импортом по шифру объекта.

### 7.3. Что откуда для отчётов (кратко)

| Потребность | Основной источник |
|-------------|-------------------|
| Статус передачи объекта (зелёный / синий / красный по бизнес-правилам) | `rooms.status_id` + название из `status` |
| ДДУ / ДКП | `deal.contract_type_id` |
| Даты АПП, договора, односторонней | `deal.*_date` |
| Число клиентских приёмок, ответственный | Агрегация по `GET /inspections` с `typeId=1` и `roomId` |
| Окно приёмки по датам | `take_date_from` / `take_date_to` у `room` или у `inspection` |
| Проект, фаза, корпус, white box, проданный объём | Лист **Импорт** (и производные), не корень API `rooms` |
| Замечания по приёмке | `inspections` с `embed=claims` |

## 11. Расширенная карта: что ещё можно получить в CRM и как

Источник: публичная [статья «Интеграция и обмен данными с внешней CRM»](https://YOUR_CRM_API_HOST/blog/19-integration-help), [описание API](https://YOUR_CRM_API_HOST/static/about-api), [вебхуки](https://YOUR_CRM_API_HOST/blog/18-webhooks).  
Страница `https://YOUR_CRM_API_HOST/api/documentation` без входа показывает только форму авторизации — **точные сигнатуры и поля тел только в Swagger после входа в ЛК**.

### 11.1. Три способа получить данные из Базис Недвижимость (CRM)

| Способ | Как | Когда удобно |
|--------|-----|----------------|
| **Опрос API по расписанию** | Bearer-токен, `GET` списков и сущностей | Пакетная выгрузка в Sheets / BI (как ваши скрипты) |
| **Webhooks** | Настройка URL в ЛК, `POST` от CRM при событии | Реакция в реальном времени (новая приёмка, смена статуса, техзаявка) |
| **Выгрузка в Excel из ЛК** | Ручной экспорт | Разовый аудит без разработки |

### 11.2. Типы данных, которые официально упоминаются как «забираемые из БН»

Из статьи: новая приёмка и дата; статус приёмки; **технические заявки**; **наряды**; отметки выполнения; изменение статуса помещения; изменение данных покупателя (ФИО, паспорт).

### 11.3. REST-методы и ресурсы (из статьи; префикс в продакшене обычно `/api/v1/` — как у вас в коде)

В статье пути иногда без префикса `api/v1` — в Swagger они должны быть полными.

| Данные | Метод / путь (как в документации статьи) | Комментарий |
|--------|------------------------------------------|-------------|
| **Токен** | `POST /oauth/token` | То же, что уже используете; в FAQ: access ~1 сутки, refresh ~30 дней |
| **Пользователи (покупатели)** | `POST /users` | Создание пользователя |
| **Паспортные данные** | `PUT /users/{userId}/personalData` | Запись ПДн |
| **Сделки** | `POST /deals` | Создание сделки с привязкой к `room_id` БН |
| **Участники сделки** | `POST /deals/{dealId}/clients` | Привязка покупателя к сделке |
| **Сделка + клиенты (чтение)** | `GET /deals/{dealId}/clients?embed=personal_data` | Список клиентов сделки, в т.ч. ПДн при `embed` |
| **Поставщик / импорт сделок** | `POST /providers/{providerId}/deals` | Upsert сделки из CRM (JSON как в фиде сделок) |
| **Карта соответствия ID** | `POST /externalObjects/{provider}/maps` | Связка ID сделки БН ↔ внешней CRM |
| **Соответствие помещений CRM ↔ БН** | `GET /externalObjects/Room` (или аналог в Swagger) | Карта внешних ID и внутренних `room_id` |
| **Приёмки за период** | `GET /inspections?dateFrom=…&dateTo=…` | В статье пример формата даты `ДД.MM.ГГГГ`; ваш код использует `takeDateFrom` / `takeDateTo` — сверить с актуальной докой |
| **Техзаявки (замечания / сервис)** | `GET /claims?dateFrom=…&dateTo=…` | Отдельная сущность от списка замечаний внутри одной инспекции (`embed=claims`) |
| **Слоты записи на приёмку по дням** | `GET /houses/{houseId}/inspectionsPeriods?groupBy=date` | Календарь доступных дней |
| **Слоты по времени на день** | `GET /houses/{houseId}/inspectionsDays/{date}/times` | `date` в примере `10.06.2021` |

В вашем коде уже есть **`GET .../inspections`** с фильтрами `houseId`, `roomId`, `typeId`, `statusId`, `embed` — это «та же семья» API, что и в статье.

### 11.4. Вебхуки (события без опроса)

Настройка: ЛК → Webhooks. В статье явно названы сценарии:

- создание / изменение **сделки**
- создание / изменение **приёмки**
- создание **техзаявки**

Полезно, когда не нужно дергать API каждую минуту и важна низкая задержка.

### 11.5. Не REST: фиды импорта (загрузка в БН, не только чтение)

| Канал | Описание | Ссылка на материал |
|-------|-----------|---------------------|
| XML-фид помещений | Дома/помещения, повторная подгрузка обновляет объекты | [Формат XML-фида](https://YOUR_CRM_API_HOST/blog/17-basis-xml-feed) |
| JSON-фид сделок | Сделки и участники | [Формат JSON сделок](https://YOUR_CRM_API_HOST/blog/22-basis-deal-json-feed) |

Это в основном путь **в БН** из CRM; обратная выгрузка из БН — через API, webhooks или настроенный JSON-фид исходящий из БН (описан в той же статье).

### 11.6. Рекомендация по «как узнать всё точно»

1. Войти в ЛК → **Настройки → Своя интеграция** — там ключи и **Swagger**.  
2. В Swagger найти ресурсы: `rooms`, `inspections`, `deals`, `claims`, `houses`, `users`, `externalObjects`, `providers`.  
3. Для одного `room_id` выполнить пробный `GET` с разными `embed` и сохранить JSON в файл — так видно реальные поля (`deal`, подписание ЭЦП и т.д.) для вашего аккаунта.

## 13. Фиды импорта и метод загрузки (публичная документация)

Источники: [Фид помещений XML/JSON](https://YOUR_CRM_API_HOST/blog/17-basis-xml-feed), [Фид сделок JSON](https://YOUR_CRM_API_HOST/blog/22-basis-deal-json-feed).

### 13.1. Загрузка помещений

| Канал | Детали |
|-------|--------|
| Форматы | **JSON** (рекомендуется, полный набор полей) или **XML** (на базе формата Яндекс.Недвижимости + доп. теги БН) |
| Идентификация при обновлении | По **вашему внешнему ID** объекта в файле — нет в БН → создаётся, есть → обновляется |
| Структура ЖК → дом → секция → этаж | Создаётся **автоматически** из вложенности фида |
| Загрузка через API | **`POST /import`** — JSON внутри поля **`data_raw`** (или бинарный файл, если JSON неудобен) |
| Загрузка без API | Меню **Объекты → Загрузить**, тип фида «Фид Базис Недвижимость JSON/XML», файл или URL |

В статье приведён **большой пример JSON-помещения**: площади (`area`, `area_fact`, …), `take_date_from/to`, `status_id`, `room_type_id`, вложенный `house` (адрес, координаты, этажность, ЖК `district`, метро), `section`, `floor`, `plan` (картинка планировки), **`custom_fields`**.

XML: справочные теги перечислены в статье; расширения относительно Яндекса — см. страницу; обязательные поля помечены в оригинале статьи.

### 13.2. Загрузка сделок (JSON-фид)

Массив сделок; внутри — **`clients`** с опциональным блоком **`personal_data`** (паспорт, адреса, ИНН, СНИЛС и т.д.).

Поля сделки из официального примера (имена полезны для сопоставления с API `deal`):

`internal_id`, `room_internal_id`, `status_id`, `contract_type_id`, `contract_number`, `contract_date`, `price`, `payments`, `surcharge_amount`, `registration_date`, `registration_number`, `act_date`, `one_sided_act_date`, `resign_one_sided_act`, `readiness_message_*`, `keys_given`, `custom_fields`, клиенты с ФИО/телефоном/`personal_data`.

Импорт через ЛК: **Объекты → Импорт → Загрузка из фида**, тип «Фид Базис Недвижимость», URL фида.
