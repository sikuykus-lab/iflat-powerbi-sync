# CRM → Google Sheets → Power BI: цепочка данных и листы

## Общий ключ

| Сущность | Первичный ключ | Связь |
|----------|----------------|--------|
| Помещение | `room.id` | Корень |
| Приёмка (клиентская) | `inspection.id` | `inspection.room_id` → `room.id` |
| Сделка | `deal.id` | `room.deal_id` или объект `room.deal` |
| Дом | `house.id` | `room.house_id` / `room.house` |
| ЖК | `district.id` | `house.district_id` при `embed=district` |

В Power BI: связь **FactПриёмки[room_id]** → **DimКомнаты[room_id]** (многие-к-одному).

## Оптимальная стратегия запросов (минимум лишних вызовов)

### 1) Лист «Комнаты» — один проход по списку помещений

**Запрос:** `GET /api/v1/rooms` с пагинацией (`page`, `perPage` до 100).

**Фильтры:** те же, что вам нужны для заселения (например несколько `statusId`: 6, 11, 12) — в OpenAPI один параметр `statusId` на запрос, поэтому **несколько статусов = несколько параллельных потоков пагинации** (как в вашем `Rooms_Refresh`), а не N запросов на каждую комнату.

**`embed` (одним списком, без второго запроса на комнату):**

`room_type,status,floor,section,house,deal,decoration,tags`

- **Тип ОН** — `room_type` + `room_type_id`
- **Статус передачи / цвет** — `status` (id, name, цвета в объекте статуса при наличии)
- **ДДУ/ДКП** — поля сделки в `deal` (`contract_type_id`, при необходимости связка со справочником типов договоров)
- **ЭЦП/бумага** — поля `deal` + при нехватке данных — `GET /deals` с `embed=contract_type,digital_documents,...` **точечно** по `deal_id`, только для строк с пробелами (см. ниже)
- **White box / отделка** — `decoration` + при необходимости `custom_fields` в `embed` у `/rooms` (тяжелее по объёму — включать при необходимости)

**Не делать:** для каждой комнаты отдельный `GET /rooms/{id}`, если первый ответ уже содержит нужные `embed`.

### 2) Лист «Приёмка» — один проход по приёмкам за период (предпочтительно)

**Запрос:** `GET /api/v1/inspections` с пагинацией.

**Фильтры:**

- `typeId` — **тип** приёмки (не статус): `1` клиентская, `2` внутренняя, `3` передача УК; в скриптах можно писать `--inspection-type client` вместо `1` (см. `iflat_inspection_labels.py`)
- `statusId` — по необходимости (можно несколько прогонов или без фильтра, если нужны все статусы)
- `takeDateFrom` / `takeDateTo` — период **проведения** приёмки (формат `ДД.ММ.ГГГГ`)
- опционально `houseId` / при необходимости сужение по объекту

**`embed` (факт приёмки + связь с комнатой и сделкой в одном ответе):**

`status,responsible,users,room,room.house,room.room_type,room.deal`

При необходимости тяжелее: добавить `steps`, `claims`, `expert` — только если эти поля реально нужны в BI (сильно увеличивают JSON).

**Почему это эффективно:** вы получаете **все строки приёмок** и **`room_id` в каждой**, без цикла «для каждой комнаты отдельный список приёмок».

### 3) Когда всё же нужны чанки `roomId`

Используйте `GET /inspections?roomId=id1,id2,...&typeId=1` только если:

- нужны **все** клиентские приёмки **за всю историю** по фиксированному списку комнат, и фильтр по дате не подходит; или
- догонка записей вне окна дат.

Тогда: батчи id (как в вашем `ROOMS_REFRESH_INSPECTION_CHUNK`), **один embed**, параллельность ограничить ради 429.

### 4) Справочники — редкие короткие GET (кэш на лист или в модели)

Вызывать **по расписанию** (раз в день/неделю), не на каждый прогон комнат:

| Данные | Запрос |
|--------|--------|
| Типы помещений | `GET /rooms/types` |
| Статусы помещений | `GET /rooms/statuses` |
| Статусы приёмок | `GET /inspections/statuses` |
| Типы договоров | `GET /deals/contractTypes` |
| ЖК | `GET /districts` |
| Дома (иерархия к фазе/секции) | `GET /houses?embed=district,sections` |

### 5) Знаменатель «% заселения от проданного объёма»

Источник «продано» в CRM: выборка помещений с **`saleStatuses=SOLD`** (`GET /rooms`, тот же `embed` по минимуму: `house,room_type,status`).

- Либо отдельный лист **«Продано_объёмы»** с зерном `room_id` (или агрегат по дому/ЖК за снимок).
- Числитель — комнаты в статусах заселения из листа «Комнаты».
- Срезы проект / фаза / корпус: через связь **DimДома** (`district`, `section` из `house`).

## Какие листы создать в дополнение к существующим

Уже есть:

- **Комнаты** — `worksheet_id` 1053493361  
- **Приёмка** — `worksheet_id` 1168786701  

Рекомендуется добавить (имена можно русифицировать; в скобках — зерно):

| Лист | Назначение | Источник API |
|------|------------|--------------|
| **Справочник_типы_помещений** | Срез «тип ОН» | `/rooms/types` |
| **Справочник_статусы_помещений** | Статусы + цвета | `/rooms/statuses` |
| **Справочник_статусы_приёмок** | Статусы клиентских приёмок | `/inspections/statuses` |
| **Справочник_типы_договоров** | ДДУ/ДКП и др. | `/deals/contractTypes` |
| **Дома** | Проект/фаза/корпус, связь с ЖК | `/houses?embed=district,sections` |
| **ЖК** | Верхний уровень локации | `/districts` |
| **Продано_помещения** (снимок) | Знаменатель для % заселения | `/rooms?saleStatuses=SOLD&...` |

Опционально:

| Лист | Когда нужен |
|------|-------------|
| **Сделки_доп** | Если в `room.deal` не хватает полей — выгрузка `GET /deals?embed=contract_type,digital_documents,room` по списку `deal_id` |
| **Календарь** | Удобно в Sheets; в Power BI чаще генерируют DAX/справочник дат в модели |
